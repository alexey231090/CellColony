# Правила проекта CellColony

## 1. Структура проекта
- `scenes/` — `.tscn` файлы (с вложенными папками по объектам).
- `scripts/` — `.gd` файлы (разделены по логике: `core`, `ai`, `resources`).
- `assets/` — Внешние ресурсы.
- `HUD/UI` — Проект использует систему «Живого интерфейса» (Living UI). Элементы управления (перки) привязаны к центру масс колонии через `PerkButtonManager`.


## 2. Работа со сценами (The Script Attachment Rule)
- **ЗАПРЕЩЕНО:** Прямая вставка пути к скрипту в ноду в `.tscn`.
- **ОБЯЗАТЕЛЬНО:** Использование `[ext_resource]` и `ExtResource(ID)`.
- **Порядок инициализации:** При спавне нод из кода (`instantiate()`), основные свойства (`owner_type`, `position`, `stats`) ДОЛЖНЫ быть установлены ДО вызова `add_child()`. Это гарантирует корректную работу `_ready()` (группы, визуализация).

## 3. Код и типизация (GDScript 2.0)
- **Строгая типизация:** Все аргументы и возвращаемые значения функций должны быть типизированы.
- **Node-модули:** Сложную логику выносить в дочерние ноды (ShooterModule, HealthModule и др.).

## 4. Специфические правила CellColony
- **Параметры:** Все базовые параметры (энергия, скорость) хранятся в `CellStats`. Однако глобальные настройки систем (например, стоимость и кулдауны перков) ОБЯЗАНЫ быть выведены как `@export` в управляющих нодах (например, в `SelectionManager`) для удобства балансировки через Инспектор. Запрещено хардкодить числа в логике или скрывать глобальные настройки внутри ресурсов каждой клетки.
- **Фракции и Группы (OwnerType):** 
  - Все клетки обязаны быть в группе `cells`.
  - Динамические группы фракций: `player_cells`, `enemy_red_cells`, `enemy_green_cells`, `enemy_yellow_cells`.
  - Переключение фракций выполняется через `_capture(new_owner: OwnerType)` с обновлением групп и ОБЯЗАТЕЛЬНЫМ сбросом всех временных баффов (щиты, множители, ускорения, эффекты вируса).
- **Мир и ИИ:**
  - Генерация мира (`main.gd`) обязана учитывать 4 базы по углам и создание **биологических стенок** (`_generate_borders()`) по периметру `map_size`.
  - Камера должна быть ограничена координатами `map_size` для предотвращения вылета за пределы уровня.
  - ИИ (`AIFactionManager`) использует систему весов (скоринг) для выбора целей по всей карте. Оптимизация ИИ-циклов обязательна (избегать вложенных поисков по всем клеткам).
  - Режим наблюдателя активируется автоматически в `CameraFollow.gd`, когда группа `player_cells` пуста.
  - **Безопасность ИИ:** Поиск своих клеток в `AIFactionManager` должен иметь запасной вариант (fallback) через фильтрацию группы `cells` по `owner_type`, если фракционная группа пуста.
  - **ИИ и Перки:** Фракции ИИ ОБЯЗАНЫ накапливать энергию перков при захвате клеток и использовать их тактически.
- **Визуализация (Rendering):**
  - Сложные визуальные эффекты (сетка сот, Rim Lighting, сложные деформации) ДОЛЖНЫ реализовываться через шейдеры (GPU), а не через программную отрисовку `_draw()`, чтобы не перегружать CPU при большом кол-ве юнитов.
- **Безопасность:** Перед работой с `target_node` (узлом-целью) всегда использовать `is_instance_valid(target_node)`.
- **Стрельба:** Снаряды должны пролетать сквозь союзников, если они не являются целью (учитывать 4 фракции).
- **Взаимодействие систем:** 
  - Связь UI и кода должна идти через группы. Пример: Поиск `SelectionManager` через группу `selection_manager` в кнопке перка.
  - Состояния выбора (например, активация перка) должны храниться в `SelectionManager.gd` и блокировать стандартное поведение клика.
  - **Перки:** При добавлении нового перка необходимо:
	 1. Определить его стоимость в энергии перков, длительность и кулдаун.
	 2. Добавить визуальную обратную связь (эффекты, звуки, анимации).
	 3. Реализовать **на-клеточную активацию**:
		- **Double-click:** Мгновенная активация по центру клетки (обычный режим).
		- **Drag-to-Aim (зажатие):** Позволяет сместить зону действия (Щит/Вирус) ИЛИ задать направление рывка (Спринт).
	 4. Добавить визуализацию статуса на саму клетку:
		- Иконка перка (индивидуальная геометрия или спрайт).
		- Радиальный индикатор кулдауна.
		- Индикация доступности через яркость/прозрачность (энергия).
	 5. Обеспечить сброс баффа при захвате клетки.
	 6. Обеспечить сброс баффа при захвате клетки ИЛИ при заражении вирусом.
	 7. Внести описание перка в документацию (`REFERENCE.md` и `project_rules.md`).


	- Прямое управление (горячие клавиши 1-4) сохранено, но кнопки больше не отображаются на краях экрана.
	- **Иммунитет к вирусу:** Клетка получает иммунитет к конкретной волне вируса (ID) после выздоровления. Повторное заражение возможно только новым «штаммом» (новым ID от ИИ или игрока).
	- **Блокировка:** Зараженная клетка теряет доступ к активации перков и визуально гасит щит.

## 5. Взаимодействие с пользователем
- Никакое изменение кода или архитектуры не выполняется без подтверждения пользователя, кроме визуальных улучшений и исправлений ошибок (по запросу).
- Всегда отвечать на Русском языке.
- Всё делается шаг за шагом с подтверждением и предложением следующего шага.
- При добавлении новых механик (например, перков) необходимо обновить документацию и проверить соответствие правилам проекта.
