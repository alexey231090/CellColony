# Правила проекта CellColony

## 1. Структура проекта
- `scenes/` — `.tscn` файлы (с вложенными папками по объектам).
- `scripts/` — `.gd` файлы (разделены по логике: `core`, `ai`, `resources`).
- `assets/` — Внешние ресурсы.
- `HUD/UI` — Весь интерфейс должен быть процедурным (`GDScript`-driven), адаптивным под ПК и мобильные устройства и выдержанным в стиле **Glassmorphism / Living UI**. Основные элементы (меню, панели, кнопки паузы) создаются программно для гибкости, четкости и удобной дальнейшей доработки.
- **Дизайн Код:** Стиль **Glassmorphism** (полупрозрачность, размытие, неоновые обводки).
- `LevelManager` подключен как autoload в `project.godot` и является центральной точкой конфигурации уровней.


## 2. Работа со сценами (The Script Attachment Rule)
- **ЗАПРЕЩЕНО:** Прямая вставка пути к скрипту в ноду в `.tscn`.
- **ОБЯЗАТЕЛЬНО:** Использование `[ext_resource]` и `ExtResource(ID)`.
- **Порядок инициализации:** При спавне нод из кода (`instantiate()`), основные свойства (`owner_type`, `position`, `stats`) ДОЛЖНЫ быть установлены ДО вызова `add_child()`. Это гарантирует корректную работу `_ready()` (группы, визуализация).

## 3. Код и типизация (GDScript 2.0)
- **Строгая типизация:** Все аргументы и возвращаемые значения функций должны быть типизированы.
- **Node-модули:** Сложную логику выносить в дочерние ноды (ShooterModule, HealthModule и др.).

## 4. Специфические правила CellColony
- **Параметры:** Все базовые параметры (энергия, скорость) хранятся в `CellStats`. Однако глобальные настройки систем (например, стоимость и кулдауны перков) ОБЯЗАНЫ быть выведены как `@export` в управляющих нодах (например, в `SelectionManager`) для удобства балансировки через Инспектор. Запрещено хардкодить числа в логике или скрывать глобальные настройки внутри ресурсов каждой клетки.
- **Фракции и Группы (OwnerType):** 
  - Все клетки обязаны быть в группе `cells`.
  - Динамические группы фракций: `player_cells`, `enemy_red_cells`, `enemy_green_cells`, `enemy_yellow_cells`.
  - Переключение фракций выполняется через `_capture(new_owner: OwnerType)` с обновлением групп и ОБЯЗАТЕЛЬНЫМ сбросом всех временных баффов (щиты, множители, ускорения, эффекты вируса).
  - **Отставшая клетка:** Если клетка ненейтральной фракции сильно оторвалась от центра своей колонии, она обязана получить состояние `stranded`: ускоренно возвращаться к центру колонии, терять по `1 HP` раз в секунду и при `0 HP` становиться нейтральной с полным сбросом движения и stranded-состояния.
- **Архитектура уровней:**
  - Единый источник истины для уровней — `res://scripts/core/LevelManager.gd`.
  - Список уровней задается только через массив `LEVELS`. Запрещено дублировать список уровней, количество уровней или `scene_path` в других местах проекта.
  - Экран выбора уровней обязан получать число уровней через `LevelManager.get_total_levels()`.
  - Переход в уровень обязан идти через `LevelManager.set_current_level(level_num)` и чтение сцены через `LevelManager.get_current_level_scene_path()`.
  - Сцены уровней (`main.gd`, `organic_level.gd` и будущие) обязаны читать параметры запуска через `LevelManager.get_current_level_data()` в `_ready()`.
  - Формат элемента `LEVELS`: `id`, `scene_path`, `num_enemies`, `has_islands`, `is_organic`, `map_scale`, `num_neutrals`, `seed` и дополнительные поля генерации по необходимости.
  - Начиная с 1 уровня, основной шаблон для новых уровней — `res://scenes/levels/organic_level.tscn`.
  - Планируемые 20-40 уровней должны в первую очередь отличаться параметрами генерации: размером арены, формой внешней границы, числом/формой/расположением островов, а не копированием новых сцен.
  - Для organic-уровней конфиг допускает параметры вида `shape_type`, `shape_size`, `shape_power`, `play_area_radius_mult`, `noise_freq`, `noise_amp`, `island_count`, `island_radius`, `island_noise_freq`, `island_noise_amp`, `island_specs` и аналогичные поля.
  - Новый уровень может быть как новой сценой, так и новым конфигом поверх существующей сцены. Если различается только генерация, обязателен новый конфиг без дублирования `.tscn`.
  - Organic-уровень обязан сам расставлять игрока, ИИ и нейтралов по конфигу. Для спавнов нужно избегать островов и стартовых зон других фракций.
  - Для уровня с островами допустим ранний физический отступ внешней стены и расширенный коллайдер острова, если это улучшает читаемость столкновений клеток.
- **Мир и ИИ:**
  - Генерация мира (`main.gd`) обязана учитывать 4 базы по углам и создание **биологических стенок** (`_generate_borders()`) по периметру `map_size`.
  - Камера должна быть ограничена координатами `map_size` для предотвращения вылета за пределы уровня.
  - ИИ (`AIFactionManager`) использует систему весов (скоринг) для выбора целей по всей карте. Оптимизация ИИ-циклов обязательна (избегать вложенных поисков по всем клеткам).
  - **Старт матча:** Нейтральные клетки должны пересоздаваться со случайными позициями на каждом запуске уровня. Нельзя делать их вечной фиксированной раскладкой, если это не отдельное явно задокументированное свойство уровня.
  - Режим наблюдателя активируется автоматически в `CameraFollow.gd`, когда группа `player_cells` пуста.
  - **Безопасность ИИ:** Поиск своих клеток в `AIFactionManager` должен иметь запасной вариант (fallback) через фильтрацию группы `cells` по `owner_type`, если фракционная группа пуста.
  - **ИИ и Перки:** Фракции ИИ ОБЯЗАНЫ накапливать энергию перков при захвате клеток и использовать их тактически.
  - **Приоритет ИИ:** До обнаружения врага в радиусе внимания ИИ должен в первую очередь забирать нейтральные клетки, а не открывать бой через всю карту.
  - **Визуализация (Rendering):**
	- **Игровой мир:** Сложные визуальные эффекты (сетка сот, Rim Lighting, эффекты скорости) ДОЛЖНЫ реализовываться через шейдеры (GPU). Шейдеры должны поддерживать наложение нескольких эффектов (например, щит + спринт одновременно).
  - **Интерфейс (UI):** Кнопки, панели и декоративные элементы интерфейса должны по умолчанию строиться процедурно через `_draw()` и/или `StyleBoxFlat`, чтобы сохранять четкость на любых разрешениях и легко адаптироваться под ПК и мобильные экраны. Запрещено использовать растровые текстуры для простых геометрических элементов.
- **Безопасность:** Перед работой с `target_node` (узлом-целью) всегда использовать `is_instance_valid(target_node)`.
- **Стрельба:** Снаряды должны пролетать сквозь союзников, если они не являются целью (учитывать 4 фракции).
- **Щит и вирус:** Щит отражает только обычные снаряды. Вирусные снаряды не должны отражаться щитом и не могут заражать через рикошет.
- **Стрельба и окружение:** Снаряды обязаны сталкиваться с биологическими стенками и островами. Visual impact по стене должен вычисляться отдельно от столкновения клеток, чтобы вспышка сидела ближе к поверхности препятствия.
- **Движение у препятствий:** Если на organic-уровне есть острова, движение клетки к цели не должно ломаться об лобовое столкновение. `MoverModule` обязан использовать систему **Dual Whisker Steering** (3 луча: центр и два «уса» под 30°) для плавного обхода препятствий с памятью направления скольжения.
- **Взаимодействие систем:** 
  - Связь UI и кода должна идти через группы. Пример: Поиск `SelectionManager` через группу `selection_manager` в RadialPerkMenu.
  - Состояния выбора (например, активация перка) должны храниться в `SelectionManager.gd` и блокировать стандартное поведение клика.
  - **Перки:** При добавлении нового перка необходимо:
	 1. Определить его стоимость в энергии перков, длительность и кулдаун.
	 2. Добавить визуальную обратную связь (эффекты, звуки, анимации).
	 3. Добавить иконку и цвет в `RadialPerkMenu.gd` (массивы `perk_names` и `perk_colors`).
	 4. Реализовать логику активации в `SelectionManager.gd` (функция `activate_perk()`).
	 5. Добавить обработку в `_is_perk_ready()` и `get_perk_cooldown_ratio()`.
	 6. Обеспечить сброс баффа при захвате клетки ИЛИ при заражении вирусом.
	 7. Внести описание перка в документацию (`REFERENCE.md` и `project_rules.md`).

- **Добавление нового уровня:** При добавлении нового уровня необходимо:
	 1. По умолчанию использовать `res://scenes/levels/organic_level.tscn` и новый конфиг в `LevelManager.LEVELS`.
	 2. Если создается новая сцена уровня, подключать ее скрипты только через `[ext_resource]` и `ExtResource(ID)`.
	 3. Добавить новый `Dictionary` в `LevelManager.LEVELS` с уникальным `id` и полным набором параметров формы, размера, островов и состава матча.
	 4. Убедиться, что сцена уровня читает конфиг через `get_current_level_data()`, а не через локальные хардкоды.
	 5. Новые варианты островов и формы сначала реализовывать через параметры (`noise_*`, `play_area_radius_mult`, `island_*`, `island_specs`), а не через отдельную сцену.
	 5.1. Если форма ближе к коробке, использовать `shape_type = "rounded_box"` вместо отдельной сцены.
	 5.2. Если уровень стартовый или компактный, отдельно балансировать `num_neutrals` и диапазон энергии нейтралов, а не копировать настройки с больших карт.
	 6. Не добавлять ручные `match`, `if level_num == ...` и другие развилки по сценам в UI, если тот же результат можно получить из `LevelManager`.
	 7. Обновить документацию, если изменился формат `LEVELS` или появились новые обязательные параметры уровня.


	- Прямое управление (горячие клавиши 1-4) сохранено для быстрого доступа к перкам.
	- **Радиальное меню:** Зажатие ПКМ (ПК) или долгий тап 0.3с (мобильные) открывает круговое меню с 4 перками. Меню появляется в позиции курсора/тапа и не загромождает экран.
	- **Иммунитет к вирусу:** Клетка получает иммунитет к конкретной волне вируса (ID) после выздоровления. Повторное заражение возможно только новым «штаммом» (новым ID от ИИ или игрока).
	- **Блокировка:** Зараженная клетка теряет доступ к активации перков, визуально гасит щит и **постепенно теряет энергию** со скоростью регенерации владельца (до 0).

## 5. UI/UX Стандарты (Yandex Games)
- **Подход:** UI обязан оставаться процедурным, а не зависеть от набора статичных растровых кнопок или панелей. Визуальный стиль: Glassmorphism / Living UI с чистой геометрией, мягким свечением и аккуратной анимацией только там, где она улучшает читаемость.
- **Адаптивность:** Весь UI обязан корректно работать на ПК (16:9) и мобильных (вертикаль/горизонталь). Использовать якоря (`Anchors`) и `CanvasLayer` слои.
- **Слои (Layering):** 
	- `HUDLayer`: 100 (игровой интерфейс)
	- `PerkPanel`: 110 (кнопки способностей)
	- `PauseButton`: 115 (мобильная кнопка паузы)
	- `PauseMenu / Overlay`: 120 (меню паузы)
- **Пауза:** Меню паузы, открываемое по `Esc` и мобильной кнопке, должно содержать как минимум действия `Продолжить`, `Рестарт` и `Выйти в меню`.
- **Анимация:** Использовать `Tween` для мягких переходов. Избегать изменения `scale` у элементов внутри `BoxContainer` (вызывает дёргание Layout). Для пульсации использовать `modulate:a` (прозрачность).

## 6. Взаимодействие с пользователем
- Никакое изменение кода или архитектуры не выполняется без подтверждения пользователя, кроме визуальных улучшений и исправлений ошибок (по запросу).
- Всегда отвечать на Русском языке.
- Всё делается шаг за шагом с подтверждением и предложением следующего шага.
- При добавлении новых механику или UI элементов необходимо обновить документацию.
